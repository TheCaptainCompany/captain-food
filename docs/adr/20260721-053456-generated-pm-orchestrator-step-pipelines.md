# ADR-20260721-053456 — Generated process-manager orchestrator step pipelines (hooks architecture)

## Status

Accepted — implements issue #25 / codegen-roadmap item 3; builds on ADR-20260719-172821 (the typed
step DSL), ADR-20260719-193500 (the state-table PM runtime), ADR-20260721-031734 (#27 generated PM
state stores) and ADR-20260721-043033 (#26 generated service-catalog ports).

## Context

ADR-20260719-193500 deliberately hand-wrote the first state-table orchestrators "until the shape is
proven". Four orchestrators then implemented their `specs/processmanager.yaml` legs against the same
conventions, and every upcoming saga (#20 connect flow, #28 delivery partner) would have added
another hand copy. The deferral condition was met: the shape is proven, and the prerequisites the
generated pipelines lean on — the `by_*`/`upsert` state stores (#27) and the `<Base>Service` ports
(#26) — are themselves generated, so the step DSL's `state.by` keys and `call` operations map 1:1
onto generated Rust surfaces.

Studying the four hand orchestrators showed the DSL is complete for the PIPELINE but deliberately
prose-bound for a small set of seams: frozen-snapshot payload assembly, derived deterministic ids,
per-aggregate fold idempotency, the bounded re-offer branch arithmetic, and envelope-owned row
fix-ups (`client_secret` clearing, ADR-20260720-015500). Roadmap item 3 anticipated exactly this
split: "generate the step pipeline …; hand-written only the non-structural guard predicates behind
generated hook traits", with computed business logic an explicit non-goal.

## Decision

A new `codegen-rs` emitter (`emit_pm_orchestrators`) generates
`crates/application/src/generated/process_managers.rs` from `specs/processmanager.yaml`: one module
per process manager, one `async fn` per `receives` leg executing the DSL's ordered typed steps.

**Generated by construction (the pipeline):**

- `state.by` lookups via the generated `by_*` store methods, with the missing-row policy typed from
  the spec: a bare `guard throws` immediately after `state.by` IS the orphan error (context key =
  the `by` column, camelCased, matching errors.yaml); on a COMMAND leg the first `that`-guard's
  error also types the missing row ("also thrown when no run exists"); otherwise absence is a benign
  event-leg skip.
- `state.expect` mismatches → benign skips carrying the spec note; `state.set` row construction
  (struct-update over the loaded row, or a full literal for opening legs: unset nullable → `None`,
  `last_update_utc` → envelope-stamped) + `upsert`.
- Structural `that` guards on `message`/`state`/read aliases (enum consts; `text` columns compare as
  strings), with `skip: true` → `Outcome::Skipped` and `throws` → the typed rejection.
- `call` steps through the generated service ports with `ServiceCallMeta::new(correlation_id)`.
- `deliver` plumbing: stream addressing by the target aggregate's key (payload property →
  state-row column → read alias, `Option` keys skip the deliver — e.g. a run with no payment
  intent), `load` → idempotency predicate → `Repository::save` under the saga actor
  (`saga_actor(env)` on event legs, the command actor otherwise). Payloads whose REQUIRED
  properties are all covered by `with` are generated struct literals (with the closed coercion set:
  optionality wraps, `Money→MoneyCents`, `i32→i64`); anything else is a builder hook.
- `send` steps invoke the target's command handler with the event-leg semantics from
  ADR-20260719-193500 §4 generated in place: version conflicts and repository errors propagate (the
  runner retries the leg), any other rejection is logged and skipped; a single send downgrades the
  leg outcome to `Skipped`, a `for_each` send skips per item.
- A **pk-admission seam** on opening legs (no `state.by` before `state.set`): the existing row is
  looked up and an `admit` hook (default: admit) may veto with a benign skip — how RefundProcess
  refuses to regress a DECIDED run.

**Hook traits (one per leg, generated signatures, hand-implemented):** `read_<alias>` (returning a
generated per-alias struct whose fields are the CONSUMED columns typed by their sinks — state
column > payload literal > guard > builder-only `Option<sink>`), `build_<event>` (payloads beyond
`with`), `input_<port>_<op>` (service-call inputs; `Skip` skips just the call),
`should_deliver_<event>` (per-aggregate fold idempotency), `admit`, `finalize` (default no-op —
the `client_secret` clearing lives here), `compute_<col>`, and `branch`.

**Two DSL-reading conventions** make the last hand nuances spec-addressable without changing the
frozen spec: a self-referential `set: { X: { from_state: X } }` marks an ORCHESTRATOR-COMPUTED value
(`compute_X` hook — the re-offer counter increment), and a mid-leg bare `guard { skip: true }` is
the LINEAR-BRANCH marker (the `branch` hook picks the re-offer or exhausted branch,
ADR-20260720-004556).

**Explicitly not generated:** the `PlaceOrder` command leg (`PM_HAND_WRITTEN_LEGS`) stays
`commands::place_order` — server-side pricing is the roadmap's stated non-goal.

The four modules under `crates/application/src/process_managers/` shrink to hook impls + thin
wrappers with UNCHANGED call surfaces, so the saga runner, the generated GraphQL mutations
(`approve_refund`/`deny_refund`) and the behaviour tests keep their existing signatures. The
behaviour suite is the parity gate and stays green; two skip-message substring assertions were
updated to the generated wording (message strings were never part of tests.yaml).

## Alternatives considered

- **A data-driven step interpreter** (steps as runtime tables) — rejected: loses compile-time
  typing of payload construction and store lookups, and hides control flow the generated code makes
  reviewable.
- **One hooks trait per PM** — rejected: every wrapper would need the union of the PM's
  dependencies (a settlement leg would demand the read repositories it never uses); per-leg traits
  keep each wrapper's dependency set exact, at the cost of a few duplicated one-line impls (shared
  via a local macro in RefundProcess).
- **Generating the non-structural guards from notes** — rejected: notes are prose by design
  (ADR-20260719-172821 kept conditions beyond `{ const }` out of the DSL deliberately).

## Consequences

### Positive
- A new saga leg is spec + small hook impls: the pipeline, skip/throw semantics, envelope stamping
  and send/deliver plumbing are uniform BY CONSTRUCTION — the divergence class the issue targeted
  is structurally gone. #20/#28 get materially cheaper.
- The generator is the executable proof that the DSL means what it says: unsupported shapes fail
  generation loudly instead of being hand-interpreted.

### Negative
- The hook surface is a new generated↔hand contract: renaming a DSL alias or event renames trait
  methods (a compile error, caught by `make rust`, but still churn).
- The admission seam does one extra pk lookup on opening legs that never override `admit`
  (cart binding, dispatch) — negligible, and uniformity keeps the emitter regular.
- `PM_HAND_WRITTEN_LEGS` and the two DSL-reading conventions (self-`from_state`, mid-leg bare
  `skip` guard) are emitter knowledge documented here and in the emitter — candidates to fold into
  the DSL proper if a second consumer appears.
