# Claude rules — C4 architecture

C4 views live as **source-managed DSL** under `specs/architecture/` and are validated like the rest of
the model. The codegen emits three views from them (ADR-0013): **Structurizr DSL**
(`specs/generated/c4.generated.dsl`), **Mermaid** (`specs/generated/c4.generated.md`), and the HTML doc's zero-dependency
**interactive drill-down map** (§13). Use the `structurizr` skill to render/export. Never hand-edit the
generated files — change the YAML and `npm run generate`.

## Files

- `specs/architecture/c4-l2.yaml` — Level 2: the `system`, `boundedContexts` (domain decomposition,
  each binding its `aggregates`/`processManagers` by `$ref` into `actors.yaml`), `containers`
  (deployable units; `realizes` actors), `externalSystems`, and `relationships`.
- `specs/architecture/c4-l3.yaml` — Level 3: `components` of the `api` container. `command-handlers`
  bind aggregates (`handles`), `projection-updaters` bind read models (`updates` → `views.yaml`), and
  each component declares `instrumented: true|false` (the observability boundary).

## Mapping guidance

- **L2 containers** = bounded contexts / deployable runtime units.
- **L3 components** = command handlers, event consumers, projections, GraphQL gateway, BAM projector,
  observability middleware, ACL, payment adapter, etc.
- **L4** is manual / tightly supervised. Runtime loops must not invent L4.

## Operation → bounded context (how the docs group the API)

The generated documentation is organized top-level by bounded context. Operation placement is **derived,
deterministic, and declared** in `c4-l2.yaml` — never guessed by role-name or context-name matching:

- A **mutation** belongs to the context of the **aggregate/process-manager that handles its command**
  (the `boundedContexts` membership). Customer-performed writes (`placeOrder`, `rateOrder`, `addCartLine`)
  therefore live in `order` (handled by `Order`/`PlaceOrderProcess`/`Cart`), not `customer`; `acceptOrder`
  shares `order` for the same reason. To move a command's context, move its handling actor's context.
- A **query/subscription** belongs to the **performer** context — the `roles:` declared on the bounded
  context (reads/discovery are caller-facing; browsing restaurants is `customer`, not `restaurant`) —
  falling back to the read model's context when its roles span several contexts (e.g. `orders`).
- **Domain artifacts** (actors, views, commands, events, entities, scalars, errors, tests, observability)
  always group by their aggregate's context.

## Consistency checks (enforced by `npm run validate`)

- Every aggregate/PM in `actors.yaml` is mapped to a bounded context (`c4-actor-unmapped` warns on
  drift).
- Every `$ref` in the C4 files resolves (no phantom container/component referencing a missing
  actor/view) — via the generic referential check.
- Bounded-context `roles` are valid `UserType` values and no role is claimed by two contexts
  (`c4-context-role-unknown` / `c4-context-role-overlap`).

## When the model changes

Add a new aggregate → put it in a bounded context (and, if it runs in the API, in `c4-l3` `handles`).
Add a new `View_*` → add it to `projection-updaters.updates`. Keep the C4 YAML in step with `actors.yaml`
and `views.yaml`; the validator will flag drift.
